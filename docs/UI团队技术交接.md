# LineSpace UI 与数据接口工作指引

LineSpace 是一个面向诗歌创作、分享和协作写作的移动端产品。当前使用一套 Expo/React Native 代码，同时支持 iOS、Android 和 Vercel 移动网页。

注意：`apps/mobile` 不是重复项目，而是主要客户端。它内部：

- `app` 目录负责页面地址和路由入口。
- `src/features` 负责真正的页面内容和交互。

当前 README 和部分说明文档可能滞后。Codex 开始工作前，应以实际源码、配置、路由和运行结果为准，不要仅根据旧文档判断项目状态。

## 核心工作原则

UI 团队可以：

- 优化现有页面和组件。
- 新增页面、交互和素材。
- 增加 Mock 演示数据。
- 在新功能需要额外数据时，新增数据类型、API 方法和 API 路由。
- 为未来后端实现预留清晰、可替换的数据接口。

但必须遵守：

- 不删除或重命名现有接口。
- 不改变现有接口的请求方式、路径和原有字段含义。
- 不把原有可选字段改成必填字段。
- 不破坏现有页面、Mock 模式和 HTTP 模式。
- 新接口必须在原有架构基础上增加，保持向后兼容。
- 页面不能直接调用 `fetch`，数据请求应统一通过 `lineSpaceApi` 和 `packages/api-client`。

“向后兼容”是指：增加新功能以后，原有页面和原有调用方式仍然可以正常工作。

## 会接触到的目录

- `apps/mobile/src/features`：首页、创作、诗歌详情、个人主页等具体页面。
- `apps/mobile/app`：页面路由入口，通常保持轻量。
- `packages/ui/src/components`：多个页面都会使用的组件。
- `packages/tokens/src`：颜色、字体、间距和圆角等设计变量。
- `apps/mobile/assets`：页面图片、背景、头像和插画。
- `packages/ui/src/icon`：可复用 SVG 图标。
- `packages/api-client/src/mock-data.ts`：前端演示数据。
- `packages/api-client/src/types.ts`：前后端共享的数据结构。
- `packages/api-client/src/client.ts`：统一 API 能力和 Mock 实现。
- `packages/api-client/src/http-client.ts`：真实 HTTP 请求实现。
- `apps/api`：后端接口入口和未来后端实现位置。

前端默认可以使用 Mock 数据，因此 UI 团队不需要等待数据库或正式后端完成。

未来接入数据库时，应替换 API 内部实现，而不是重新改写页面组件。

## 推荐素材目录

```text
apps/mobile/assets/
├─ images/
├─ backgrounds/
├─ avatars/
└─ illustrations/

packages/ui/src/icon/
└─ 可复用 SVG 图标
```

文件使用小写英文和连字符，例如：

```text
poem-paper-background.png
profile-default-avatar.png
compose-empty-state.svg
```

素材规则：

- 不要把整张页面截图作为实际界面。
- 不要使用临时占位图替代 Figma 已提供的真实素材。
- 不要为了少量图标新增一整套图标依赖。
- 页面专用素材放在 `apps/mobile/assets`。
- 多个页面复用的 SVG 图标放在 `packages/ui/src/icon`。
- 删除旧素材前，必须先检查它是否仍被代码引用。

## Figma 与 Codex 的协作方式

设计师完成页面后，需要复制具体 Frame、组件或 Variant 的 Figma 链接，而不是只复制整个 Figma 文件链接。链接最好包含 `node-id`。

需要同时告诉 Codex：

- Figma 节点链接。
- 对应的目标页面。
- 要保留的现有交互。
- 页面需要展示的数据。
- 是否需要新增数据或接口。
- 手机和移动网页的适配要求。

Codex 应按照以下顺序工作：

1. 获取目标节点的 design context。
2. 如果节点过大，先读取 metadata，再读取具体子节点。
3. 获取同一节点的 screenshot。
4. 获取 variables，包括颜色、字体、间距和圆角。
5. 获取设计中使用的图片和 SVG。
6. 检查项目中是否已有对应组件和 Token。
7. 实现页面。
8. 对照 Figma 截图进行视觉检查。
9. 验证手机和移动网页尺寸。

如果 Figma MCP 无法使用，只能完成结构实现，不能声称已经达到 1:1 视觉还原。

## 页面需要新数据时如何处理

先判断属于哪种情况。

### 情况一：只是临时展示内容

例如推荐诗歌、作者名字或演示评论。

可以先在以下位置增加 Mock 数据：

```text
packages/api-client/src/mock-data.ts
```

不要把大量演示内容直接写进页面组件。

### 情况二：现有接口已经提供所需数据

直接复用现有 `lineSpaceApi` 方法，不要重复创建相似接口。

### 情况三：确实需要新的后端数据

允许新增接口，但必须完整增加以下链路：

1. 在 `packages/api-client/src/types.ts` 增加新的类型或可选字段。
2. 在 `LineSpaceApi` 中增加新的方法。
3. 在 Mock API 中实现该方法。
4. 在 `HttpLineSpaceApi` 中实现对应 HTTP 请求。
5. 在 `apps/api` 中增加对应 API 路由。
6. 在 `mock-data.ts` 中补充演示数据。
7. 页面通过 `lineSpaceApi` 使用该数据。
8. 记录未来数据库需要实现的内容。

例如，一个新的“诗歌推荐理由”功能可以增加：

```text
GET /v1/poems/:id/recommendation
```

但不能为了这个功能随意改变现有的：

```text
GET /v1/poems/:id
```

如果只需要给现有返回对象增加信息，优先增加可选字段，保证旧页面仍然可以处理没有该字段的数据。

## 新增接口时的兼容规则

允许：

- 新增 API 路径。
- 新增 `LineSpaceApi` 方法。
- 新增数据类型。
- 为现有类型增加可选字段。
- 新增 Mock 数据。
- 新增后端路由占位实现。
- 新增错误状态和加载状态。

默认不允许：

- 删除现有 API。
- 重命名现有 API。
- 修改现有 API 路径。
- 改变现有字段的含义。
- 将现有可选字段改成必填。
- 让 Mock API 和 HTTP API 返回不同结构。
- 在页面中绕过 `lineSpaceApi` 直接请求后端。
- 将服务端密钥放入前端。
- 未经确认直接执行数据库迁移或修改线上数据库。

如果新功能需要修改数据库，可以增加独立的 Schema 或迁移草案，但不要直接操作正式数据库。必须在交付报告中说明未来后端要完成的工作。

## 每个页面的实施步骤

1. 在 Figma 中确定最终 Frame 和组件状态。
2. 明确页面需要的数据和交互。
3. 整理图片、字体、图标及命名。
4. 提供精确的 Figma 节点链接。
5. 让 Codex 检查现有组件、Token 和接口。
6. 使用 Codex 按节点实现页面。
7. 优先复用现有接口；确实缺少数据时，再增加兼容的新接口。
8. 在 Mock 模式下完成页面预览。
9. 对照 Figma 检查字体、间距、颜色、滚动和点击区域。
10. 检查手机和移动网页尺寸。
11. 运行类型检查与生产构建。
12. 汇报新增接口、Mock 数据和未来后端工作。

## 后端能力的保留与扩展

登录、数据持久化、多人实时协作、图片上传、AI 辅助写诗和内容审核等功能，可以在 UI 阶段先定义数据需求和接口契约。

UI 团队可以新增：

- 共享类型。
- Mock 数据。
- API 客户端方法。
- 后端路由骨架。
- 请求和返回参数校验。
- 加载、成功、空状态和错误状态。

UI 团队不应擅自完成：

- 正式身份认证方案。
- 线上数据库迁移。
- 服务端密钥配置。
- 真实 OpenAI 调用。
- 生产环境存储和权限策略。
- 实时协作冲突处理。
- 线上限流和内容审核策略。

这些内容应在交付报告中列为后续后端任务。

## 工作原则

当前 README 和 docs 可能滞后。先检查实际源码、路由、组件、类型和 API 实现，再开始修改。

必须保留：

- 现有页面路由。
- 现有交互行为。
- 现有 API 路径和方法。
- 现有数据字段含义。
- Mock 模式。
- HTTP API 模式。
- 手机端和移动网页能力。

可以在新功能确实需要数据时，增加新的类型、Mock 数据、API 方法和后端路由，但不得破坏现有接口。

## Figma MCP 流程

1. 读取目标节点的 design context。
2. 如果内容过大或被截断，读取 metadata，并拆分读取具体节点。
3. 获取同一节点的 screenshot。
4. 获取 variables，包括颜色、字体、间距、圆角和效果。
5. 获取真实图片和 SVG 素材。
6. 检查现有组件、Token 和 Code Connect 映射。
7. 完成实现后，对照截图检查视觉一致性。

如果 Figma MCP 不可用，停止宣称精准还原，只完成可明确判断的结构工作，并汇报限制。

## UI 实现要求

- 页面实现优先放在 `apps/mobile/src/features`。
- 路由入口保持在 `apps/mobile/app`，且尽量轻量。
- 可复用组件放在 `packages/ui/src/components`。
- 设计变量放在 `packages/tokens/src`。
- 页面素材放在 `apps/mobile/assets`。
- 可复用 SVG 放在 `packages/ui/src/icon`。
- 优先复用已有组件和 Token。
- 不使用整页截图充当界面。
- 不为了图标新增大型依赖。
- 保留现有导航、点击、编辑和数据行为。
- 补充加载、空数据、失败和重试状态。
- 检查手机安全区、滚动、键盘遮挡和移动网页宽度。

## 数据与接口要求

先检查现有 `LineSpaceApi` 是否已经提供所需数据。

如果现有接口可以满足：

- 直接复用。
- 不创建功能重复的新接口。
- 不在页面中直接使用 `fetch`。

如果确实需要新增数据：

1. 在 `packages/api-client/src/types.ts` 增加清晰的数据类型。
2. 优先使用新的可选字段，避免破坏旧数据。
3. 在 `LineSpaceApi` 中增加新方法。
4. 在 Mock API 中实现新方法。
5. 在 `HttpLineSpaceApi` 中实现同样的方法。
6. 在 `apps/api` 中增加新的兼容路由。
7. 在 `mock-data.ts` 中提供可用于 UI 预览的数据。
8. 页面只能通过 `lineSpaceApi` 使用该能力。
9. 记录未来数据库、认证、存储或 AI 服务需要完成的实现。

新增接口时禁止：

- 删除、重命名现有接口。
- 修改现有 API 路径。
- 改变现有字段的含义。
- 将已有可选字段改成必填。
- 让 Mock 实现与 HTTP 实现返回不同结构。
- 在前端加入服务端密钥。
- 未经要求执行真实数据库迁移。
- 为了 UI 方便而绕过统一 API 层。

## 验证要求

完成后至少检查：
- TypeScript 类型检查。
- Web 生产构建。
- 目标页面在 Mock 模式下可以显示。
- 原有路由仍然有效。
- 原有页面仍可使用。
- 新增接口的 Mock 与 HTTP 类型一致。
- 手机尺寸和移动网页尺寸。
- Figma 截图与实际页面的主要视觉差异。
